索引-性能分析-慢查询日志

发表于2024-12-11|更新于2024-12-13|MySQL
|总字数:451|阅读时长:1分钟|浏览量:

想要优化SQL语句，我们首先要知道是哪个步骤执行的频率更高

可以看到主要是查询为主

可以看到慢查询日志默认状态是关的

关闭MySQL再次打开，已经是打开状态

但这种状态只是临时的，当关闭MySQL再打开的时候，慢查询日志还是关闭状态

设置了慢查询日志的路径之后，会生成慢查询日志的log文档

Linux系统中cat+日志地址可以查看日志基本信息（数据库版本，端口…）

当超出慢查询时间的阈值的时候，会显示出信息，包括使用的数据库、操作时间、操作语句

Linux系统开启慢查询实时监测

因为Windows没有tail命令，所以不能直接执行

set long_query_time = -1不加global的解释

SET timestamp是一个时间戳，而不是一个执行时间

上一篇
索引-最左前缀法则
意思是只要最左边的字段存在与条件之中，放哪里都不会影响索引 可以看到gender、age、profession三个字段建立了联合索引，并且有顺序的 我们正常去查询的时候，是会用到索引的，对应上之前创建的索引idx_gender_age_profession（gender，age，profession对应着1，2，3的顺序） 如果我们只根据gender和age查询，也是会用到索引的，但此时的key_len已经发生变化，说明profess索引长度为42 - 9 = 33 如果最左边的gender没有出现，则索引不会生效（不满足最左前缀法则） 如果中间跳了某些字段，那么该字段后面的字段索引也会失效，比如上图的key_len为4，但profession为33，说明profession的索引失效了 如果索引序号为1的字段不紧跟where之后，那么所有索引不会失效 如果age使用了>或<的其中一个，那么其右侧的索引全部失效，这里的右边不是查询语句里的右边，而是索引需要的右边，当前age为2，那么为3的profession就失效了

索引-结构
没有索引的查询要把整张表扫描一遍，有索引的查询明显更高效 指针比key值多1 B-tree只有在一层的结点数等于最大度数-1之后才会中间值（偶数偏右）向上分裂 567在变成叶子结点之后，链表中依然存在 B+tree相对于B-tree的区别：所有数据都会出现在叶子节点、叶子节点形成一个单链表 单链表之上的全是某个时间段的中间结点 一个叶子结点存储在页当中，一页为16KB 如果出现哈希碰撞，则需要到链表中去查询 键就是该层数据上一层的中间结点，由于B-tree每一层都存放键值对，导致需要更多的空间，而B+tree一般非叶子结点只会放键
